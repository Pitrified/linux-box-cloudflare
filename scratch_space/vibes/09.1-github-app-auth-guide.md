# GitHub App Authentication Guide
## Programmatic Repo Access Without PATs

---

## Overview

A **GitHub App** is the production-grade alternative to Personal Access Tokens for automating repository access. Unlike PATs - where the repo list is baked in at creation time and requires a new token whenever you expand scope - a GitHub App decouples _what permissions exist_ (defined once at app creation) from _which repos are accessible_ (managed dynamically via installation settings). Adding a new repo to the perimeter is a one-click operation in the GitHub UI; no code or credentials change.

The credential model is asymmetric and short-lived:

- You store a **private key** (`.pem` file) that never leaves your machine.
- At runtime, you sign a short-lived **JWT** (10 min max) with that key.
- You exchange the JWT for an **installation access token** (1 hour), which behaves exactly like a Bearer token or a git credential.

This guide covers the complete flow: one-time app creation, installation, token generation in Python, and ongoing maintenance.

## Part 1 - One-Time Setup

### Step 1: Create the App

Navigate to **github.com/settings/apps/new**.

#### Required fields

| Field | What to fill in |
|---|---|
| **GitHub App name** | Any unique name, e.g. `my-homelab-bot` |
| **Homepage URL** | Required but cosmetic for private apps - use `https://github.com/yourusername` |

#### Sections to skip entirely

The form has several sections that are irrelevant for a private automation app:

- **Identifying and authorizing users** - leave the Callback URL blank; uncheck "Expire user authorization tokens"; leave "Request user authorization during installation" unchecked; leave "Enable Device Flow" unchecked. These are all for OAuth flows where human users log in through your app.
- **Post installation** - leave Setup URL blank; leave "Redirect on update" unchecked.
- **Webhook** - uncheck the **Active** checkbox. You don't need webhooks for a token-generation use case. The Webhook URL field will disappear once you uncheck Active.
- **Subscribe to events** - leave everything unchecked.

#### Permissions

Expand the **Repository permissions** section. For each resource, the dropdown offers: No access / Read-only / Read and write.

Set only what you actually need. For a typical CI/automation use case (read code, push commits, create releases):

| Permission | Recommended level |
|---|---|
| **Contents** | Read and write - covers commits, branches, releases, downloads |
| **Metadata** | Read-only - **mandatory**, automatically set, cannot be changed |
| **Pull requests** | Read and write - if you need to open or merge PRs |
| **Actions** | Read and write - if you need to trigger or read workflow runs |
| **Workflows** | Read and write - if you need to update `.github/workflows` files |
| **Checks** | Read and write - if you post CI status checks |
| **Commit statuses** | Read and write - legacy status API alternative to Checks |

Everything else (Codespaces, Dependabot, Pages, Packages, etc.) - leave at **No access** unless you have a specific need. The **Organization permissions** and **Account permissions** sections can be left entirely at No access for a repo-scoped automation.

#### Where can this App be installed?

Select **Only on this account**. This keeps the app private and prevents anyone else from installing it.

#### Submit

Click **Create GitHub App**. You will land on the app's settings page.

### Step 2: Generate a Private Key

On the app settings page, scroll to the bottom to the **Private keys** section and click **Generate a private key**. A `.pem` file will download automatically.

Store it somewhere stable on your machine, e.g. `~/.config/github-apps/my-homelab-bot.pem`. Set permissions to `600`.

```bash
chmod 600 ~/.config/github-apps/my-homelab-bot.pem
```

### Step 3: Note Your App ID

On the same app settings page, near the top, you'll see **App ID** - a small integer like `12345678`. Note it down; you'll hardcode it or put it in a config file.

### Step 4: Install the App

The app exists but has no access to anything yet. To grant it access to repos:

1. On the app settings page, click **Install App** in the left sidebar.
2. Click **Install** next to your account.
3. Choose **Only select repositories** and pick the repos you want this app to have access to.
4. Click **Install**.

After installation, look at the URL - it will be something like `github.com/settings/installations/12345678`. That number at the end is your **Installation ID**. Note it down.

Alternatively, find it via the API:

```bash
# Sign a JWT first (see Part 2), then:
curl -H "Authorization: Bearer <jwt>" \
     -H "Accept: application/vnd.github+json" \
     https://api.github.com/app/installations
```

## Part 2 - Token Generation

Install the required Python library:

```bash
uv add PyJWT cryptography httpx
# or: pip install PyJWT cryptography httpx
```

### The Token Helper

```python
import time
from pathlib import Path

import httpx
import jwt  # PyJWT


def get_installation_token(
    app_id: int,
    private_key_path: str | Path,
    installation_id: int,
    repositories: list[str] | None = None,
) -> str:
    """
    Generate a GitHub App installation access token.

    Args:
        app_id: The App ID from the app settings page.
        private_key_path: Path to the downloaded .pem file.
        installation_id: The installation ID for your account.
        repositories: Optional list of repo names (without owner) to further
                      restrict the token scope. If None, covers all repos
                      in the installation.

    Returns:
        A Bearer token valid for 1 hour.
    """
    private_key = Path(private_key_path).read_text()

    # Build and sign a JWT valid for 9 minutes (GitHub max is 10)
    now = int(time.time())
    payload = {
        "iat": now - 60,   # issued slightly in the past to account for clock skew
        "exp": now + 540,  # 9 minutes from now
        "iss": str(app_id),
    }
    jwt_token = jwt.encode(payload, private_key, algorithm="RS256")

    # Exchange the JWT for an installation token
    url = f"https://api.github.com/app/installations/{installation_id}/access_tokens"
    headers = {
        "Authorization": f"Bearer {jwt_token}",
        "Accept": "application/vnd.github+json",
        "X-GitHub-Api-Version": "2022-11-28",
    }

    body = {}
    if repositories:
        body["repositories"] = repositories

    response = httpx.post(url, headers=headers, json=body or None)
    response.raise_for_status()

    data = response.json()
    return data["token"]


# --- Usage ---

APP_ID = 12345678          # from app settings page
INSTALLATION_ID = 99999999 # from installation URL or API
PEM_PATH = "~/.config/github-apps/my-homelab-bot.pem"

# Token covering all installed repos
token = get_installation_token(APP_ID, PEM_PATH, INSTALLATION_ID)

# Token restricted to a specific repo (subset of the installation)
token = get_installation_token(
    APP_ID, PEM_PATH, INSTALLATION_ID,
    repositories=["my-repo"]
)
```

### Using the Token

The token is a standard Bearer token. Use it anywhere a PAT would be accepted:

```python
# GitHub REST API
headers = {"Authorization": f"Bearer {token}"}
resp = httpx.get("https://api.github.com/repos/you/my-repo/contents/", headers=headers)

# Git over HTTPS
# Set the remote URL to:
#   https://x-access-token:<token>@github.com/you/my-repo.git
import subprocess
subprocess.run([
    "git", "clone",
    f"https://x-access-token:{token}@github.com/you/my-repo.git"
])
```

### Integrating With Your Release Pipeline

Since `update_git_deps.py` already orchestrates repo operations, the cleanest integration is a thin wrapper that lazily fetches and caches the token:

```python
import time
from functools import cached_property
from dataclasses import dataclass


@dataclass
class GitHubAppAuth:
    app_id: int
    installation_id: int
    pem_path: str

    _token: str | None = None
    _token_expiry: float = 0.0

    def token(self) -> str:
        """Return a valid token, refreshing 5 minutes before expiry."""
        if time.time() < self._token_expiry - 300:
            return self._token

        self._token = get_installation_token(
            self.app_id, self.pem_path, self.installation_id
        )
        self._token_expiry = time.time() + 3600
        return self._token
```

## Part 3 - Adding Repos (Day-2 Operations)

When you create a new repo and want the app to have access to it:

1. Go to **github.com/settings/installations**.
2. Click **Configure** next to your app.
3. Under **Repository access**, click the repo selector and add the new repo.
4. Click **Save**.

That's it. No key rotation, no new token generation setup, no config file changes. The next call to `get_installation_token()` will automatically include the new repo.

## Part 4 - Maintenance

### Private Key Rotation

You can have multiple active private keys simultaneously. To rotate:

1. Go to the app settings page → **Private keys** → **Generate a private key**.
2. Download the new `.pem` and update `PEM_PATH` in your config.
3. Delete the old key from the GitHub UI once confirmed working.

There is no forced expiry on private keys - rotate on your own schedule or if you suspect compromise.

### Modifying Permissions

If you later need to add a new permission (e.g. you want the app to also manage Issues):

1. Go to the app settings page → **Permissions & events** → adjust the relevant permission dropdown.
2. Save. GitHub will notify you that the installation needs to accept the updated permissions.
3. Go to **github.com/settings/installations** → **Configure** → review and accept the new permissions.

Note: reducing permissions takes effect immediately; adding permissions requires the installation-acceptance step.

### Revoking Access to a Repo

Go to **github.com/settings/installations** → **Configure** → remove the repo from the selector → Save. Tokens minted after this change will not include that repo; existing tokens expire within 1 hour anyway.

### Deleting the App

If you ever want to decommission the whole thing:

1. Go to the app settings page → scroll to the bottom → **Delete GitHub App**.
2. All installations are automatically revoked.
3. Delete the `.pem` file from your machine.

## Part 5 - Reference: Key Identifiers

Keep these three values in a config file or environment variables:

| Identifier | Where to find it | Example |
|---|---|---|
| `APP_ID` | App settings page, "About" section | `12345678` |
| `INSTALLATION_ID` | URL of the installation settings page, or `GET /app/installations` | `99999999` |
| `PEM_PATH` | Local path to the downloaded private key | `~/.config/github-apps/my-homelab-bot.pem` |

## Part 6 - Security Summary

| Property | Detail |
|---|---|
| **What's stored** | Private key (`.pem`) - never transmitted directly |
| **What travels over the wire** | Signed JWTs (10 min lifetime) and installation tokens (1 hour lifetime) |
| **Blast radius of a leaked token** | Limited to repos in the installation, limited to declared permissions, expires in ≤1 hour |
| **Blast radius of a leaked private key** | Can mint new tokens until key is revoked from the app settings page |
| **Audit trail** | Actions show up in GitHub audit log attributed to the app, separate from human actions |
| **Revocation** | Key: revoke from app settings page; Token: expires naturally or delete the installation |
