## Plan: Fix hatchling direct-reference rejection

**Root cause:** hatchling refuses to build a package whose `[project.optional-dependencies]`
(or `[project.dependencies]`) contain PEP 440 direct URL references (`@ git+https://...`)
unless the project explicitly opts in with:

```toml
[tool.hatch.metadata]
allow-direct-references = true
```

This is a hatchling-specific gate - uv resolves direct references fine, but hatchling validates
metadata at build time and rejects them by default.

**Affected repos:**
- `media-downloader` - direct refs in `[project.optional-dependencies]` (the reported error)
- `laife` - direct ref in `[project.dependencies]`; same build backend, same rule applies

### Step 1 - Fix media-downloader

In `media-downloader/pyproject.toml`, add after `[tool.hatch.build.targets.wheel]`:

```toml
[tool.hatch.metadata]
allow-direct-references = true
```

### Step 2 - Fix laife

Same addition in `laife/pyproject.toml`.

### Step 3 - Verify and commit both

```bash
cd /home/pmn/repos/media-downloader && uv sync --all-extras --all-groups
# cd /home/pmn/repos/laife && uv sync # laife sync deferred to later date
```

Commit each repo separately:

```bash
git add pyproject.toml && git commit -m "fix: allow direct url references in hatchling metadata"
```

### Notes

1. This setting is safe - it only unlocks validation, it does not change install behaviour.
2. `llm-core` and `fastapi-tools` are leaf libraries (no direct refs in their own deps) so they
   are unaffected.
3. The `allow-direct-references` block should be added to the **library release checklist** in
   `linux-box-cloudflare/docs/git-tag-libraries.md` so future libraries with git-tag deps do not
   hit this again.
